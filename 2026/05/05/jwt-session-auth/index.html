Session 和 JWT：两种登录状态保存方式

发表于2026-05-05|更新于2026-05-05|技术
|浏览量:

登录认证的核心问题是：用户登录成功之后，服务端如何在后续请求中识别这个用户。常见方案主要有两类：基于服务端状态的 Session，以及基于令牌签名的 JWT。

Session 验证方式

Session 的基本流程是：

  1. 用户首次登录成功后，服务器随机生成一个 sessionId。
  2. 服务器通过 Set-Cookie 响应头让浏览器保存这个 sessionId。
  3. 浏览器后续请求会自动携带 Cookie。
  4. 服务器从 Cookie 中取出 sessionId，再去 Redis 或内存中查询对应的用户信息。
  5. 如果能查到有效 Session，就认为用户已经登录。

需要注意的是，服务端不应该盲目信任 JWT Header 中传来的算法，而应该使用后端配置中允许的算法进行校验。

Session 的优缺点

Session 的优点是服务端可控性强。因为登录状态存储在服务端，所以如果用户退出登录、账号被封禁、密码被修改，服务端可以直接删除或更新对应的 Session，让登录状态立即失效。

Session 的缺点是需要服务端保存状态。在单机系统中，Session 可以直接放在内存中；但在分布式部署下，多个后端实例需要共享 Session，因此通常会引入 Redis 作为集中式 Session 存储。

引入 Redis 后可以解决多实例之间的 Session 共享问题，但也会带来新的成本：

  • Redis 容量需要规划。
  • Session 过期策略需要设计。
  • Redis 高可用需要维护。
  • 每次请求都可能产生一次 Redis 网络访问。
  • 用户量过大时，Session 数据会增加 Redis 内存压力。

所以 Session 更适合传统 Web 应用。App 和小程序也可以使用类似 sessionId 的机制，但它们不像浏览器一样天然依赖 Cookie，因此实际项目中更常使用 Token 方案。

JWT 的优缺点

JWT 的优点是服务端不需要保存用户登录状态。只要服务端能验证签名，就能判断令牌是否可信。这种方式更适合前后端分离、App、小程序以及多端登录场景。

JWT 的缺点是令牌一旦签发，在过期之前通常会一直有效。如果 Access Token 泄露，攻击者可以在令牌过期前冒用用户身份。由于 JWT 本身不依赖服务端状态，服务端很难让某一个已经签发的 JWT 立刻失效，除非额外引入黑名单、版本号或集中式存储。

Access Token 和 Refresh Token

为了解决 JWT 泄露后风险窗口过长的问题，实际项目中通常会把令牌拆成两类：

类型 作用 有效期
Access Token 调用业务接口时验证用户身份 较短
Refresh Token Access Token 过期后用于刷新令牌 较长

Access Token 的有效期一般比较短，例如 15 分钟、30 分钟或 2 小时。它用于访问普通业务接口。

Refresh Token 的有效期一般更长，例如 7 天或 30 天。它只用于刷新 Access Token，不应该频繁暴露给普通业务接口。为了更好地控制风险，Refresh Token 通常可以存放在 Redis 中，这样服务端可以主动删除它，从而让用户重新登录。

不过需要明确一点：Refresh Token 不能让已经泄露的 Access Token 立刻失效。它的主要作用是让 Access Token 的有效期可以设置得更短，从而缩短泄露后的风险窗口。

Cookie 的角色

Cookie 不是一种登录认证方案，而是浏览器保存和自动携带数据的一种机制。

Cookie 可以用来保存 sessionId，也可以用来保存 JWT。区别在于：

  • 保存 sessionId 时，真正的登录状态在服务端。
  • 保存 JWT 时，登录状态主要由令牌签名和过期时间来证明。

所以讨论登录认证时，不应该把 Cookie 和 Session、JWT 放在同一个层级比较。Cookie 是一种浏览器存储和传输机制，Session 和 JWT 才是认证方案。

总结

Session 和 JWT 没有绝对的优劣，关键是看业务场景。

如果系统是传统 Web 应用，并且希望服务端能强控制登录状态，Session 是更直接的方案。如果系统需要支持 App、小程序、多端接入或前后端分离，JWT 会更灵活。

在实际项目中，也可以组合使用：用短期 Access Token 验证接口访问，用长期 Refresh Token 刷新令牌，并把 Refresh Token 存在 Redis 中统一管理。这样可以在性能、扩展性和安全性之间取得更好的平衡。
